# AI Collaboration Operating System

面向小项目、初创团队、黑客松和早期产品的轻量 AI-native 协作操作系统。

它不是一套大而全的工程流程，也不是把成熟公司内部 AI 基建缩小后照搬到小项目里。它的目标是帮助一个项目快速建立最小但有效的协作边界：AI 应该读什么、哪些规则更权威、当前阶段不能做什么、用户行为如何形成闭环、完成工作要拿什么证据，以及当 AI 或人类协作偏航时如何纠正。

## 为什么需要它

很多先进的 AI-native 开发经验来自成熟项目和大型团队。那些经验很有价值，但往往依附于完整的测试体系、PR 流程、平台工程、代码所有权、事故复盘和长期组织记忆。

小项目、初创团队和黑客松项目通常没有这些基础设施，也不应该为了协作而先引入沉重流程。它们更需要一套可以快速落地、逐步增强、能阻止常见 AI 协作失败的轻量系统。

这个项目关注的问题包括：

- AI 读了太少上下文就开始改代码。
- 本地草稿、旧审计或截图被误当成正式规则。
- 项目一次性过度开发，提前引入数据库、登录、AI 服务、CI 或测试框架。
- 用户界面显示“成功”，但没有任何 store、service、API 或持久化写入。
- 文档规则重复出现在多处，其中一处修改后其他地方没有同步。
- AI 跳过验证，却自信地宣称工作完成。
- 用户纠正后，只道歉不改变下一步行动。

## 核心原则

AI-native 项目文档不是知识堆积，而是针对重复 agent 失败模式的控制系统。

每条规则都应该回答一个问题：它在防止哪一种协作失败？

## P0 包含什么

当前版本是一个最小草稿，只包含：

```txt
skills/
  project-collaboration-operating-system/
    SKILL.md
README.md
```

其中 `project-collaboration-operating-system` skill 用于重新设计或改进一个项目的 AI 协作体系，重点覆盖权威上下文、阶段边界、行为闭环、本地草稿、验证契约、重复文档漂移和纠偏循环。

## 如何使用

如果你的 AI 工具支持 skills，可以把这个目录中的 skill 复制到对应的 skills 目录中：

```txt
skills/project-collaboration-operating-system/
```

然后在需要改造项目协作体系时，让 agent 使用这个 skill，例如：

```txt
Use project-collaboration-operating-system to diagnose and improve this project's AI collaboration docs.
```

如果你的工具暂时不支持 skills，也可以直接阅读 `SKILL.md`，把它作为协作体系审计和重构清单使用。

## 适合谁

- 正在从 demo 走向 MVP 的小项目。
- 需要多人和 AI 一起开发的早期产品。
- 黑客松、课程项目、个人开源项目。
- 想引入 AI 协作，但不想一开始就背上复杂工程流程的团队。

## 不适合什么

- 已有完整平台工程、治理流程和内部 AI 基建的大型组织。
- 需要替代测试、代码审查、CI 或安全审计的场景。
- 希望一次生成完整项目管理体系的模板化需求。

这个项目提供的是轻量起点，不是万能流程。

## 当前状态

这是早期 P0 版本。后续可能继续增加：

- references：更详细的权威模型、重复文档审计、行为闭环和验证契约说明。
- scripts：用于发现重复规则、文档漂移和协作风险的轻量脚本。
- templates：可复制的 `AGENTS.md`、协作说明、编码规范和决策记录模板。
- examples：来自真实小项目的失败模式和改进案例。

当前阶段先保持最小结构，方便复制、迁移和继续迭代。
